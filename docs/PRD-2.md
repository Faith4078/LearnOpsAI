# PRD 2: LearnOps AI — Customer Education Operations Upgrade

**Status:** ready-for-agent
**Source:** docs/client-brief.md ("Additional client updates")
**Date:** 2026-07-07
**Parent:** docs/PRD.md (original scope, issues #1–#8, implemented)

## Problem Statement

The MVP works, but it reads like an AI content generator: you paste docs, AI writes, you publish. The client's Customer Education team needs it to feel and function like an internal knowledge operations platform — where content is a governed asset with quality scores, review reports, educational metadata, and lifecycle visibility — not an AI playground. Operators can't see quality at a glance (no review report or scores), articles carry no educational framing (objectives, difficulty, audience), there is no governance trail on published assets, and the landing page doesn't communicate operational status.

## Solution

Extend — without rewriting — the existing pipeline into a Customer Education Operations platform. The Generator Agent additionally produces educational metadata; the Review Agent additionally produces a structured quality-assurance report (score, changes made, publishing recommendation) shown as a new Quality Assurance Report stage on the timeline with per-stage timings. Published knowledge assets carry their metadata and governance record into Sanity and display them on enterprise-style article pages with a governance sidebar. The Home page opens with a compact Knowledge Operations Dashboard (live stats where computable, realistic placeholders otherwise) and an architecture panel explaining the automation pipeline. All product language shifts to knowledge-operations vocabulary. The two-AI-call constraint is preserved: metadata rides in the generator's single response, the QA report rides in the reviewer's single response.

## User Stories

1. As a content operator, I want the app to present itself as a Customer Education Operations platform, so that stakeholders perceive an enterprise tool rather than an AI toy.
2. As a content operator, I want the Generator Agent to produce learning objectives alongside the article, so that readers know what an asset teaches.
3. As a content operator, I want estimated reading time generated for each asset, so that readers can plan their learning.
4. As a content operator, I want a difficulty level assigned to each asset, so that content matches learner skill levels.
5. As a content operator, I want a target audience identified for each asset, so that assets reach the right readers.
6. As a content operator, I want prerequisites listed for each asset, so that learners prepare before reading.
7. As a content operator, I want a Quality Assurance Report after the Review Agent runs, so that I can judge publish-readiness at a glance.
8. As a content operator, I want an overall quality score in the review report, so that quality is quantified, not vibes.
9. As a content operator, I want the report to itemize the changes the Review Agent made (clarity, formatting, duplicate removal, quiz fixes), so that I can trust and audit the AI's editing.
10. As a content operator, I want a readability assessment in the report, so that I know the content suits its audience.
11. As a content operator, I want an explicit publishing recommendation (e.g. Ready to Publish / Needs Attention), so that the go/no-go decision is guided.
12. As a content operator, I want a Quality Assurance Report stage on the workflow timeline, so that QA is a visible step in the pipeline.
13. As a content operator, I want each completed timeline stage to show how long it took (e.g. "Completed in 4.2s"), so that I can gauge pipeline performance.
14. As a content operator, I want the published Sanity document to carry the review score, metadata, and review record, so that governance data survives publishing.
15. As an operations lead, I want a Knowledge Operations Dashboard on the landing page (published assets, drafts, pending review, average review score, average processing time, knowledge base health, last published), so that I see the state of the knowledge base immediately.
16. As an operations lead, I want dashboard figures computed from live Sanity data wherever possible, so that the numbers are real, with realistic placeholders clearly reasonable where live data doesn't exist.
17. As an operations lead, I want an architecture panel on the dashboard describing each pipeline stage's responsibility, so that new team members understand the automation at a glance.
18. As a help-center visitor, I want learning objectives displayed on every article, so that I know what I'll learn before reading.
19. As a help-center visitor, I want a difficulty badge and reading time on articles and listing cards, so that I can pick content that fits me.
20. As a help-center visitor, I want a governance sidebar on each article (status, review score, freshness score, generated by, review agent version, documentation version, last reviewed, published date), so that I can trust the content's provenance and currency.
21. As a help-center visitor, I want the article layout to resemble an enterprise documentation platform, so that the content feels authoritative.
22. As a stakeholder, I want knowledge-operations terminology everywhere (Knowledge Assets, Knowledge Base, Publishing Workflow, Content Governance, AI Review) and no "AI writer/magic/chat/assistant" language, so that the product reads as enterprise software.
23. As a developer, I want the existing architecture, folder structure, design system, and seams preserved, so that the extension does not destabilize working features.
24. As a developer, I want the metadata and QA report carried inside the existing two AI calls, so that the call-count constraint and free-tier economics hold.

## Implementation Decisions

- **No rewrites.** All changes extend the existing modules: the AI schemas module, the two agent actions, the pipeline hook, the timeline component, the Sanity schema/mapper, and the Help Center pages. Folder structure, design language, and the provider/Sanity injection seams are unchanged.
- **Still exactly two AI calls.** The generator prompt/schema grows an `educationalMetadata` object (learningObjectives: string[], estimatedReadingMinutes: number, difficulty: "beginner" | "intermediate" | "advanced", targetAudience: string, prerequisites: string[]). The reviewer prompt/schema grows a `reviewReport` object returned alongside the improved bundle (overallQualityScore 0–100, readability assessment, changesMade: categorized strings covering clarity/formatting/duplicates/quiz, publishingRecommendation: "ready" | "needs-attention"). Zod schemas are extended in the single canonical schema module; both remain validated on parse.
- **Backward compatibility of the bundle contract:** the content bundle type gains the metadata field as required for new runs; the reviewer consumes and returns the full extended bundle. The review action's result type carries `{ bundle, report }`.
- **Per-stage timing** is measured client-side in the pipeline hook (wall-clock around each action call) and stored per stage; the timeline renders "Completed in Xs" per finished stage. A new "Quality Assurance Report" stage sits between Review Agent and Publishing — it represents presenting the report (no extra AI call); it completes when the report renders.
- **Sanity `helpArticle` schema extension:** new fields for educational metadata (objectives, readingMinutes, difficulty, audience, prerequisites) and governance (reviewScore, publishingRecommendation, generatedBy — model identifier, reviewAgentVersion — a version string constant, documentationVersion — a hash/label of the source input, lastReviewedAt). Publish maps all of it. GROQ projections extended accordingly.
- **Knowledge Operations Dashboard:** a server-rendered stats row on Home fed by a stats query over Sanity (published count, average review score, last published date — live; drafts/pending review/average processing time/health — derived where possible, otherwise realistic static placeholders clearly encapsulated in one placeholder module so they're easy to replace later).
- **Architecture panel:** a static presentational component on Home listing the five pipeline stages with one-line responsibility descriptions.
- **Terminology sweep:** UI copy across shell, forms, cards, toasts, and pages moves to knowledge-operations vocabulary; no code identifiers need renaming (avoid churn), only user-facing strings.
- **Governance sidebar** on the article page renders the stored governance fields; freshness score is computed presentational logic from publishedAt/lastReviewedAt age (no new storage).
- **Design:** harvey.ai-design.md remains the single source of truth; new components reuse existing card/badge/typography patterns.

## Testing Decisions

- Same testing philosophy and seams as PRD 1 — external behavior through the actions layer with `setModelCaller` / `setSanityWriter` / `setSanityReader` fakes; no network, no prompt-wording assertions. Prior art: the existing tests directory (generate/review/publish/fetch/quiz suites).
- Extended-schema coverage: generateContent returns validated educational metadata; reviewContent returns improved bundle + validated review report; missing/invalid metadata or report → invalid-response.
- Publish coverage: extended mapping writes metadata + governance fields (asserted through the fake writer).
- Stats coverage: the dashboard stats function computes live figures from a fake reader and merges placeholders deterministically.
- Freshness/reading-time style derivations are pure functions with direct unit tests (pattern established by the quiz logic module).
- Per-stage timing display and copy changes are presentational — not unit-tested, consistent with the no-UI-tests decision.

## Out of Scope

- Real review-score history/analytics (dashboard averages beyond what current documents store)
- Draft persistence (a "Draft Assets" count without stored drafts stays a placeholder)
- Human multi-step approval workflows, roles, auth
- Scheduled re-review, semantic search, version-aware regeneration (still future improvements)
- Migrating previously published articles to the new schema (new fields render as absent/graceful on old documents)

## Further Notes

- Old documents published before this upgrade lack metadata/governance fields; all display components must render gracefully when fields are missing rather than requiring a migration.
- "Review Agent Version" is a deliberate constant (e.g. "review-agent/v1") bumped when the reviewer prompt materially changes — it exists to make governance visible, not to version prompts automatically.
- The QA Report timeline stage must not introduce a third AI call; it is presentation of data already returned by the Review Agent.
